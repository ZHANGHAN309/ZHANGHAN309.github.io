西门子s7-1200
工控软件学习之路 2026/1/5

一、 核心三步循环扫描（S7-1200 的核心工作流程）

PLC 上电后，会周而复始地执行以下三步，这个循环的时间极短（毫秒级），所以你感觉设备的反应是 “实时” 的。

  1. 输入采样阶段（读信号）
    S7-1200 会扫描所有输入端子（比如接按钮的 I0.0、接传感器的 I0.1），把外部元件的状态（“按下 / 松开”“有信号 / 无信号”）读取到 PLC 内部的输入映像区。举个例子：按钮按下时，I0.0 的映像区就会变成 “1”；按钮松开，就变成 “0”。这个阶段有个关键特点：一旦采样完成，就算外部信号变化，映像区的数据也不会变，要等下一个循环才会更新。
  2. 程序执行阶段（算逻辑）
    S7-1200 会按照你写的程序（梯形图、指令表等），从上到下、从左到右逐一执行逻辑判断。执行时，它只看输入映像区的数据，不直接读外部端子。举个例子：你写的程序是 “如果 I0.0=1，就把 Q0.0 设为 1”，那么 PLC 就会查 I0.0 的映像区，满足条件就把 Q0.0 的输出映像区设为 1。
  3. 输出刷新阶段（发指令）
    程序执行完后，S7-1200 会把输出映像区的所有数据，一次性刷新到输出端子上，驱动外部设备动作。还是上面的例子：Q0.0 的映像区是 1，就会让输出端子通电，进而点亮指示灯、启动电机。刷新完成后，PLC 就会回到第一步，开始下一个循环。

二、 S7-1200 的两个辅助工作环节

除了核心三步，它还有两个重要的辅助流程，保证运行稳定：

  1. 自诊断环节每个循环开始前，S7-1200 会先自检：检查自身硬件是否故障、程序有没有语法错误、通讯是否正常。如果检测到问题（比如接线短路、程序出错），会立刻点亮面板上的故障灯，还能把故障信息上传到上位机。
  2. 通讯处理环节扫描循环的间隙，S7-1200 会处理通讯请求：比如和上位机（C# WinForm/WPF）交换数据、和其他 PLC 或触摸屏通讯。这个环节不影响核心的三步循环，保证控制逻辑的实时性。

三、 新手好记的总结

S7-1200 的工作原理 = 循环扫描三步走（输入采样 → 程序执行 → 输出刷新） + 自检 + 通讯辅助

如何搭建博客-具体流程
博客网站 2025/12/18

小记一下，在此之前一直是采用 github 托管 hexo 博客，但是确实是访问速度太慢了，便想着优化一下博客网站的搭建方式，正好今天学有所成，便来显摆(^V^)

选择博客类型

网上有许多类型的博客，我比较省钱，不想买服务器，就选择了 hexo+github page 托管的方式进行搭建，后来发现 github page 不挂梯子速度太慢了，查了相关资料选择了 netlify 托管，free 账户每个月 300 额度，相当于每个月可以上传 20 次博客，但是访问另算，所以还是要少上传，访问的话我本身不打算博客有多少人访问，纯自己写写；其他博客的类型嘛，回头补充吧–2025.12.18

搭建方式

省流版：

  1. 配置本地环境(Node.js,Git)
  2. 安装 hexo
  3. github page 或者 netlify 托管

详细版：

因为我只试过这一种搭建方式，就便只说这一种 (其实 AI 这么发达，随便搜一下也能查到)

首先需要安装本地环境，hexo 是基于 Node.js 生成的，所以需要安装 Node.js 工具，版本的话我的是 v24.10.0，npm 版本是 11.6.1，npm 应该是安装 Node.js 时附带的

这个安装完成后，需要再安装一个 git 工具用来博客版本控制，方便管理远程仓库，这个官网下载安装即可

其次下载一个文档编辑器，可以用 typora，也可以用 VSCode，我用的是 VSCode，因为可以使用 git 插件，不用手动通过命令行同步

在 github 构建一个仓库，仓库名为账号.github.io，至少我是这样的

这些都弄好之后，在 VSCode 打开 cmd 控制台，输入

npm install -g hexo-cli

选择一个文件夹作为博客根目录，后续的写博客工作就在这个目录下进行

hexo init

安装依赖包

npm install

之后该目录下面会生成 hexo 的文件，你可以从网上下载主题或者根据文档对 hexo 博客进行修改
hexo-themes不止这一个网站有主题，可以试着自己找找

接下来是上传到 github，首先在 github 创建一个仓库

  1. github page
    名字必须为用户名.github.io，其他无所谓，在仓库的 Settings-Pages-Source 里选择 Master 分支，保存后 Github 会生成访问地址，设置完成后，需要给 hexo 设置 git 插件
npm install hexo-deployer-git --save

编辑 hexo 配置文件_config.yml，修改 deploy 部分

deploy:
  type: git
  repo: https://github.com/你的用户名/你的用户名.github.io.git
  branch: master

觉得慢可以使用 ssh，但是需要生成 ssh 密钥，这个网上教程很多

deploy:
  type: git
  repo: git@github.com:账号/账号.github.io.git
  branch: master
  1. 托管

这个我熟啊，两种托管方式，一种是通过 github actions，设置管道，每次仓库更新时，自动触发逻辑处理，最终还是通过 github page 展示，启动速度慢的问题还是没解决，所以着重讲第二种方式

  • netlify 托管

去 netlify 官网绑定 github 账号，选择创建好的仓库，在 Build-Settings 里设置
Build command : hexo clean && hexo generate
Publish directory: public

之后仓库更新时，netlify 会自动更新静态网页

最近看了一下，netlify 又把免费的服务器地址设置到了 US，无法白嫖，现在托管到腾讯云的 EdgeOne，唯一的缺点是需要设置域名才能正常访问
–2025.12.26

操作步骤

在本地控制台输入

hexo clean # 清除旧静态文件
hexo g  # 生成静态文件（g = generate）
hexo s  # 启动本地服务器（s = server）
hexo d # 部署到服务器 (d = deploy)

!image

    所有的项目文件都在这里
  2. 属性（Properties）
    在这图片里没展示，基于你选择的条目可以改变他们的属性

在 XAML 代码编辑器里可以查看刚创建的项目代码

        xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
        xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
        xmlns:d="http://schemas.microsoft.com/expression/blend/2008"

        mc:Ignorable="d"
        Title="MainWindow" Height="450" Width="800">
    <Grid>

    </Grid>
</Window>

XAML 简单来说就是由 WPF 创建的 XML 语言，用来编辑页面，在<Windows>其中有三部分值得关注

  • XML 命名空间
    核心作用是避免元素命名冲突，限定文件中可使用的内容范围，无前缀的 xmlns 是默认命名空间（如 WPF 内置控件），带前缀的命名空间（如 x:、local:）用于引入额外类型（如 XAML 特性、自定义代码），xmlns:local 专门用来关联你自己项目中的代码，让 XAML 能直接使用自定义类 / 控件
  • x:Class
    x:Class属性是 WPF 中 “界面（XAML）” 和 “逻辑（代码）” 的连接核心，它明确了 XAML 对应的后台代码类，让两者能够协同工作（比如界面触发事件、代码修改界面显示）
  • Title
    窗口的标题

在 WPF 项目里 App.xaml 相当于应用程序本身，虽然它本身有内容，但是却无法通过设计器展示出来，而项目创建时会默认创建一个窗口 MainWindow.xaml，而 App.xaml 是通过头部的 StartupUri 指向展示的第一个窗口

<Application x:Class="com_mc_test.App"
             xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
             xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
             xmlns:local="clr-namespace:com_mc_test"
             StartupUri="MainWindow.xaml">

之后就是我的一些项目经验杂谈了（看不看无所谓）

写程序之前要搞清楚架构，自顶向下思考，程序是干嘛的，可以拆分哪几步，每一步需要什么功能，底层数据源从哪来，数据源怎么传递给处理层，处理层怎么传递给显示层，处理层显示层又可以分为几步，比如处理层先要解析包，再转换为某种数据类型，加密或者解密，筛分或者增删改，显示层数据是同步还是刷新，一个页面展示还是多个页面展示（感觉没说到点上）

参考了一下腾讯云[十毛]的见解，补充一下架构理解：

要写好一个项目首先要理解需求，其次再提出架构解决问题，需求是由客户或者公司提出，而架构是由架构师分析需求得出，同时因为需求可能会发生变动，所以架构师必须考虑尽可能多的情况

理解一个项目的架构首先得了解项目背景，比如这个项目是为了解决什么问题，目标用户是谁?目标用户的使用情况如何？

其次再考虑开发的功能，这些功能解决了哪些问题，核心功能有哪些，次要功能有哪些

然后列出架构图，模块层次图，核心流程图，技术点

比如说采用分层架构，表示层、业务逻辑层、数据访问层、数据库层
又比如模块，上位机模块可以分为主界面模块、日志模块、协议解析模块、插件模块、配置设计模块
核心流程图可以展示用户操作时的关键步骤，这个参考 uml
技术点可以列举前端、后端、数据库、消息队列、通信协议等

事件描述

事件相当于产生一个动作或者是某个对象状态发生改变，比如点击按钮就是一个事件。当事件发生后会触发某种信号，并提供某种事件处理方式，之后就会按照预先设定好的事件处理方式进行。

怎么使用事件

addEventListener()

const btn = document.querySelector("button");

function random(number) {
  return Math.floor(Math.random() * (number + 1));
}

function changeBackground() {
  const rndCol = `rgb(${random(255)}, ${random(255)}, ${random(255)})`;
  document.body.style.backgroundColor = rndCol;
}

btn.addEventListener("click", changeBackground);

用这样的方式可以为某个事件添加多个处理函数

btn.addEventListener("click", function1);
btn.addEventListener("click", function2);
  • removeEventListener()可以删除已经添加的事件监听器
btn.removeEventListener("click", function1);
  • 通过传递AbortSignal到addEventListener()，在拥有AbortSignal的控制器上调用abort()，从而删除事件处理器
const controller = new AbortController();

btn.addEventListener("click",
  () => {
    const rndCol = `rgb(${random(255)}, ${random(255)}, ${random(255)})`;
    document.body.style.backgroundColor = rndCol;
  },
  { signal: controller.signal } // 向该处理器传递 AbortSignal
);

controller.abort();

事件处理器属性

推荐使用addEventListener()来注册事件处理器，但是也有其他方法

const btn = document.querySelector("button");

function random(number) {
  return Math.floor(Math.random() * (number + 1));
}

btn.onclick = () => {
  const rndCol = `rgb(${random(255)}, ${random(255)}, ${random(255)})`;
  document.body.style.backgroundColor = rndCol;
};

类似

const btn = document.querySelector("button");

function random(number) {
  return Math.floor(Math.random() * (number + 1));
}

function bgChange() {
  const rndCol = `rgb(${random(255)}, ${random(255)}, ${random(255)})`;
  document.body.style.backgroundColor = rndCol;
}

btn.onclick = bgChange;

但是这种方式添加的处理器会被下一个函数覆盖

内联事件处理器–不推荐

<button onclick="bgChange()">按下我</button>

这种方式添加的处理器会不方便管理

事件对象

有时候在事件处理函数内部，可能会看到一个固定指定名称的参数，例如event、evt或e，这被称为事件对象，它会被自动传递给事件处理函数作为第一个参数。

const btn = document.querySelector("button");

function random(number) {
  return Math.floor(Math.random() * (number + 1));
}

function bgChange(e) {
  const rndCol = `rgb(${random(255)}, ${random(255)}, ${random(255)})`;
  e.target.style.backgroundColor = rndCol;
  console.log(e);
}

btn.addEventListener("click", bgChange);

事件对象的核心属性

属性 描述
type 事件类型（如 click、mouseover、keydown）。
target 触发事件的原始元素（事件最初发生的元素）。
currentTarget 绑定事件处理函数的当前元素（与 this 指向一致）。
bubbles 布尔值，指示事件是否会冒泡（向上传播）。
cancelable 布尔值，指示事件是否可以取消默认行为（通过 preventDefault()）。
clientX/clientY 鼠标事件中，鼠标相对于浏览器可视区域的坐标（不含滚动条偏移）。
pageX/pageY 鼠标事件中，鼠标相对于文档页面的坐标（含滚动条偏移）。
key 键盘事件中，按下的键对应的字符（如 a、Enter）。
code 键盘事件中，按下的物理键的编码（如 KeyA、Enter）。

事件对象的常用方法

方法 描述
preventDefault() 取消事件的默认行为（如阻止链接跳转、表单提交、右键菜单弹出等）。
stopPropagation() 阻止事件冒泡（阻止事件向父元素传播，避免父元素的事件处理函数被触发）。
stopImmediatePropagation() 不仅阻止事件冒泡，还会阻止当前元素上后续的事件处理函数执行。

示例1：阻止默认行为

<!-- 阻止链接跳转 -->
<a href="https://example.com" id="link">点击不跳转</a>

<script>
  const link = document.getElementById("link");
  link.addEventListener("click", function(e) {
    e.preventDefault(); // 取消默认跳转行为
    console.log("链接被点击，但不会跳转");
  });
</script>

示例2：阻止事件冒泡

<div id="parent" style="padding: 50px; background: lightgray;">
  父元素
  <button id="child">子元素</button>
</div>

<script>
  const parent = document.getElementById("parent");
  const child = document.getElementById("child");
  
  child.addEventListener("click", function(e) {
    e.stopPropagation(); // 阻止事件冒泡到父元素
    console.log("子元素被点击");
  });
  
  parent.addEventListener("click", function() {
    console.log("父元素被点击"); // 若不阻止冒泡，点击子元素时会同时触发此函数
  });
</script>

在事件冒泡中，事件是按照从里面的元素冒泡而出

<body>
  <div id="container">
    <button>点我！</button>
  </div>
  <pre id="output"></pre>
</body>

const output = document.querySelector("#output");
function handleClick(e) {
  output.textContent += `你在 ${e.currentTarget.tagName} 元素上进行了点击\n`;
}

const container = document.querySelector("#container");
const button = document.querySelector("button");

document.body.addEventListener("click", handleClick);
container.addEventListener("click", handleClick);
button.addEventListener("click", handleClick);

你在 BUTTON 元素上进行了点击
你在 DIV 元素上进行了点击
你在 BODY 元素上进行了点击

事件捕获可以让事件先在最小嵌套元素上发生，然后在连续更多的嵌套元素上发生，直到达到目标。

事件捕获默认是禁用的，你需要在addEventListener()的capture选项中启用它

<body>
  <div id="container">
    <button>点我！</button>
  </div>
  <pre id="output"></pre>
</body>

const output = document.querySelector("#output");
function handleClick(e) {
  output.textContent += `你在 ${e.currentTarget.tagName} 元素上进行了点击\n`;
}

const container = document.querySelector("#container");
const button = document.querySelector("button");

document.body.addEventListener("click", handleClick, { capture: true });
container.addEventListener("click", handleClick, { capture: true });
button.addEventListener("click", handleClick);

你在 BODY 元素上进行了点击
你在 DIV 元素上进行了点击
你在 BUTTON 元素上进行了点击

变量提升
技术篇 2025/11/12

var 声明，无论它们出现在脚本中的什么位置，都会在执行脚本中的任何代码之前进行处理。在代码中的任何位置声明变量都相当于在顶部声明它。这也意味着变量可以在其声明之前被使用。这种行为被称为提升，因为变量声明似乎被移动到发生该行为的函数、静态初始化块或脚本源代码的顶部。

只有变量的声明被提升，而变量的初始化则不会被提升。

function doSomething() {
  console.log(bar); // undefined
  var bar = 111;
  console.log(bar); // 111
}

类似

function doSomething() {
  var bar;
  console.log(bar); // undefined
  bar = 111;
  console.log(bar); // 111
}
